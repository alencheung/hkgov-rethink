# Next-Feature Integration Maps (post-v7)

> Generated by 4 parallel `Explore` agents (read-only research). These maps
> locate the exact integration points for the next 4 features in the PM
> strategy, so sequential implementation is de-risked. They are **research, not
> code** — each points at the file:line to change and the struct shapes to add.
>
> All four maps agree on one cross-cutting finding: **P-108 (Identity Tier) is
> the hard blocker** for server-side state in P-102/P-104/P-105. See §5 below.

---

## 🚨 Critical findings (read first — these change the plan)

The agents surfaced three load-bearing issues that any next implementation
must address:

1. **`detect_threshold_crossing` is dead code from the scheduler's POV.**
   The detector exists (`analysis.rs:886`), is tested, and is even consumed by
   P-101's Cite-It threshold-recovery — but it is **NOT wired into
   `scheduler.rs::run_one_target`'s match arm** (scheduler.rs:200-294) nor the
   `RunDetectorTool` enum (tools.rs:408). This is a hard prerequisite for
   **P-102 Signal Subscriptions** — its flagship use case ("tell me when HIBOR
   breaks 2.5%") compiles to `threshold_crossing`. Wiring it is small and
   self-contained: add the match arm + a `direction` field to `ScanTarget`.

2. **`ScanTarget` has no `direction` field** (config.rs:248-298). The
   `threshold_crossing` detector needs a `CrossDirection` (Above/Below). Either
   add `pub direction: Option<CrossDirection>` or default to `Above`. One-line
   structural change to the shared config struct.

3. **Alert dedup is global, not per-user** (alerts.rs:104, 182-200). For
   multi-user signals (P-102), the same finding must fire for user A's signal
   AND user B's independently. The dedup key must become `(signal_id,
   insight_id)`. Behavioral change, localized to `AlertDispatcher::evaluate`.

These three are the cheapest, highest-leverage fixes to land next — they
unblock P-102 without touching identity.

## P-104 — Insight Lifeline (persistence + read/unread + evolution)

**The one non-negotiable server-side piece:** evolution *detection itself* must
happen in `InsightStore::upsert` (`insight.rs:89-92`), NOT in the dashboard —
because the scheduler overwrites the prior version before any client sees it.
If the diff isn't captured at upsert time, the history is gone forever.

**Change list:**
| File | Line | Change |
|---|---|---|
| `crates/agent/src/insight.rs` | 39-66 | Add `first_seen`, `version: u32`, `evolution: Option<EvolutionDiff>` to `Insight` (all `#[serde(default)]`) |
| `crates/agent/src/insight.rs` | 89-92 | **Evolution hook**: before overwrite, hash prior content; if changed, push to history, bump version, attach `EvolutionDiff` |
| `crates/agent/src/scheduler.rs` | 135-152 | Consume the new `InsightChange` return; only feed new+evolved to alerting |
| `crates/api/src/routes.rs` | 46 | Add `/insights/{id}`, `/insights/{id}/history`, `/insights/{id}/read`, `/insights/unread` |
| `crates/api/src/state.rs` | 16 | Add `read_state: Arc<ReadStateStore>` (keyed by `X-Reader-Id` pseudo-identity) |
| `dashboard/index.html` | 183, 231-261 | localStorage read-state (`hkgov.read`, `hkgov.lastVisit`, `hkgov.watch`); `insightCard` gains unread accent + `🔔 Watch` + evolution diff block |

**Identity split:** read/unread + "what's new since last visit" + watch list
work **client-side (localStorage)** for v1 — per-browser, per-device. Cross-device
sync + server-driven delta filtering need P-108. The `X-Reader-Id` UUID bridge
(client-generated, localStorage) gives pseudo-identity matching the current
shared-key trust model.

**`EvolutionDiff` shape** (computed field-by-field in `upsert`):
```rust
pub struct EvolutionDiff {
    pub insight_id: String,
    pub from_version: u32, pub to_version: u32,
    pub from_generated_at: DateTime<Utc>, pub to_generated_at: DateTime<Utc>,
    pub changes: Vec<FieldChange>,  // Severity/Confidence/Title/Summary/Evidence deltas
}
```

## §5 — The identity layer (P-108), the shared blocker

All four maps independently confirm: **no identity layer exists.** Zero hits
for `user|session|identity|tenant|principal|subscriber` in any `*.rs` source
file. The only auth is one global shared `X-API-Key` (`auth.rs:17-43`).

**What each feature needs from identity:**
| Feature | Client-side v1 (no identity) | Needs P-108 |
|---|---|---|
| P-104 Lifeline | read/unread, "what's new", watch (localStorage) | cross-device sync, server delta filtering |
| P-102 Signals | author, preview, compile (stateless) | per-user push, per-user dedup, channel secrets |
| P-105 Drill-In | single-tab resume, share-via-URL | per-user ownership, ACLs, team cases |
| P-106 Bilingual | (none — ships independently) | (none) |

**Recommended next sequencing** (informed by these maps):
1. **Wire `threshold_crossing` into the scheduler** (critical fix #1) — unblocks P-102's flagship use case; small, self-contained.
2. **P-106 Bilingual** — no identity dep, pure localization, ships independently.
3. **P-108 Identity Tier** — the unblocker; then P-102/P-104/P-105 server-side state lands in a wave.
4. **P-102/P-104/P-105** — once P-108 is in, these three compose (they all want the same `Arc<SomeStore>` + owner principal pattern).
