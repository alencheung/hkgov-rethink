# Phase 1 — Feature Discovery & Role Mapping (canonical spreadsheet)

> Independent QA audit, Phase 1 output. Generated by reading the actual code in
> `crates/*`, `dashboard/index.html`, and `python/`. This is the single canonical
> feature inventory; Phases 2–6 reference these IDs.
>
> **Tech stack (discovered, not provided):**
> - **Backend:** Rust 2021 (edition, toolchain 1.96) — workspace of 6 crates
>   (`common`, `connectors`, `store`, `ingest`, `agent`, `api`). HTTP via
>   `axum 0.8` + `tokio`; middleware via `tower` / `tower-http`; cache via
>   `moka`; config via `figment` (TOML + env). Optional `redis` / `pg` / `llm` /
>   `alerts` cargo features.
> - **Frontend:** single static `dashboard/index.html` (vanilla JS, no build
>   step), embedded into the binary via `include_str!`.
> - **Data sources (upstream):** HKMA REST API, data.gov.hk v2 filter +
>   historical archive, HK press releases (HTML/RSS), LandsD/CSDI open tiles.
>   No SQL DB in the default path — the store is an in-process `moka` cache.
> - **Client SDK:** `hkgov-py` (Python ≥3.9, typed dataclasses, `requests`).
> - **Persistence tier:** *none by default* — every store (records, insights,
>   signals, investigations, users, feedback, alert log) is in-process
>   `Arc<RwLock<…>>`, volatile. Process restart = full data loss (by design for v1).
>
> **User roles (discovered):** there is **no multi-role RBAC system**. The auth
> model is a single optional shared API key (`X-API-Key` / `?api_key=`) that
> gates *all* non-health routes identically. A separate, orthogonal P-108
> identity tier (email + magic-link → bearer session) identifies a *user* for
> per-user features but grants **no authorization** — any authenticated user can
> read/write any owner's signals/investigations (owner is a filter, not an ACL).
> So the "roles" map to **authentication states**, not permission tiers.

## Role / authentication-state matrix

| Role ID | Authentication state | What it can reach |
|---------|----------------------|-------------------|
| **R1 — Anonymous** | No API key, no bearer | `GET /`, `GET /health`, `GET /health/sources`, `GET /v1/health/sources`, `GET /dashboard`, `GET /dashboard/` (the static HTML). **Nothing else.** |
| **R2 — Keyed caller** | Valid `X-API-Key` (or `?api_key=`), no bearer session | Every read + write route *except* the P-108 user-scoped ones resolve to `owner=""`. Can create/list/delete *any* signal/investigation (owner filter is opt-in, not enforced). |
| **R3 — Identified user** | Valid API key **and** a P-108 bearer session | Same as R2 plus the ability to pass a real `owner`/`X-Reader-Id`. `GET /v1/auth/me` resolves to a `User`. Still no ACL — can still touch other owners' resources by guessing/filtering. |
| **R4 — Admin / operator** | Not a runtime role — the person who edits `config.toml` + sets `[[agent.scan]]`, `[alerts]`, `[api.api_key]`, and rebuilds. | All of the above + the deterministic scheduler, alert dispatch, and LLM client selection. Exercised only via config + build, not via HTTP. |

> **Security note surfaced for Phase 3:** R3 has no isolation. `list_signals`
> with `owner=""` returns everyone's signals; `delete_signal`/`update_signal`
> take no owner check. This is documented as "the shared-key trust model" in the
> code comments, so it's *intended for v1* — but it's a defect-grade gap for any
> multi-tenant deployment and is tracked as a risk, not silently accepted.
